# VISION — House Electricals

> A personal-use breaker-panel mapper. The owner defines their electrical panel(s), enters each breaker, links breakers to electrical components (outlets, lights, appliances, switches), and **draws the floor plan of every floor of their home directly in the app**. Components sit on the drawn plan as touch-sized icons. Clicking any component instantly shows which breaker controls it; clicking a breaker shows everything it controls. **The user only ever interacts with the app on their phone** — the UI is a mobile-first PWA designed to feel like a modern, polished SaaS product. The whole app is shipped as a Docker Compose stack and deployed behind the user's existing reverse proxy (or a Cloudflare Tunnel).

## Problem

Homeowners don't have a reliable way to know which breaker controls which outlet/light/appliance in their house. Existing panel directories are paper labels that go stale, get illegible, or were never written down accurately. The result: trial-and-error breaker flipping, accidental power loss, or unsafe work on circuits that were never confirmed dead. House Electricals gives one homeowner one durable, searchable, visual source of truth for their electrical map — accessible from the phone they already carry, with a UI that doesn't feel like a side project.

## Goals (the only things meta-ralph is allowed to advance)

### Core goals (shipped in cycles 1–10)

- **G1 — Panel + breakers** (shipped): create panels and add breakers with slot, amperage, pole count, label.
- **G2 — Component catalog** (shipped): outlets, lights, switches, appliances, junction boxes, smoke detectors, other.
- **G3 — Breaker-to-component mapping** (shipped): each component assigned to one breaker; reverse view per-breaker.
- **G4 — House map** (shipped, baseline): place components onto a 2D floor plan with draggable, click-targetable touch-sized icons.
- **G5 — Click-to-identify** (shipped): tap component → breaker callout; tap breaker → list its components.
- **G6 — Durable server-side storage** (shipped): PostgreSQL (migrated from SQLite — see CLAUDE.md "Persistence is PostgreSQL"); relational data in a named Docker volume, floor-plan images + `.auth-secret` in a bind-mounted directory. Backup = `pg_dump` + a tar of the bind-mount.
- **G7 — Breaker-test workflow** (shipped): walk-up "flip a breaker, tag what just lost power" mode.
- **G8 — Mobile-first PWA** (shipped, baseline): installable on iOS Safari + Android Chrome; touch hit targets ≥44px.
- **G9 — LAN-only access** (shipped): no public internet required. Single-user login gate added in feat/auth-gate — first-time sign-up + scrypt-hashed password stored in Postgres; details in CLAUDE.md "Auth gate (feat/auth-gate + sign-up flow)".
- **G10 — `docker compose up` deployment** (shipped): single compose stack; one host port; HTTPS terminated by the operator's external reverse proxy (Caddy / Cloudflare Tunnel) — matches the user's HousesTracker deployment pattern.

### New goals (post cycle-10 vision extension)

- **G11 — Modern SaaS-grade UI & UX:** Replace the cycle-3 utilitarian styling with a coherent, polished design system. Concrete deliverables:
  - A token-based design system (colors, typography scale, spacing scale, elevation/shadow, radius, motion timing) defined in CSS variables.
  - Every existing screen refactored to use the system: PanelList, PanelDetail, ComponentsScreen, TestPanelScreen, PanelMapScreen.
  - **Bottom-tab navigation** for mobile (Panels / Components / Map). Top-bar replaced with contextual headers.
  - **Empty states** with illustration + call-to-action wherever a list can be empty.
  - **Loading skeletons** instead of "Loading…" text.
  - **Toast notifications** for transient feedback (saved, deleted, error). Optimistic-write errors surface as toasts, not banners.
  - **Polished iconography** (one consistent line-icon set, currently inline SVGs — pick a single visual language).
  - **WCAG AA** color contrast minimums verified.
  - **Subtle motion**: 150–250ms ease-out for entry/exit, no jarring transitions; respects `prefers-reduced-motion`.
  - The end result should feel like a 2024-era consumer SaaS product (e.g., Linear / Things / Notion polish level), not a hacker prototype.

- **G12 — In-app floor-plan editor (vector, not bitmap):** Instead of (or in addition to) uploading an image, the user draws their floor plan inside the app:
  - **Walls** are line segments with endpoints in normalized coordinates. Tools: draw wall (tap two endpoints), move endpoint, delete wall.
  - **Rooms** are named rectangles (or polygons) drawn by dragging corners. Each room has a free-text label rendered in the center.
  - **Snap-to-grid** at a configurable grid size (default 1/40 of canvas dimension) so walls connect cleanly.
  - **Vector storage**: walls and rooms persist as structured rows in Postgres, NOT as a rasterized image. They remain editable forever.
  - The existing image-upload path (G4 baseline) coexists — a floor can have a drawn plan, an uploaded image background, or both stacked (image as background, walls on top).
  - **Touch + mouse parity**: drag, two-finger zoom, two-finger pan all work; mouse + wheel zoom works on desktop dev viewport.

- **G13 — Multiple floors per home:** Currently each panel has at most one floor plan (image). Replace with house-level floors:
  - User creates floors (e.g. "Basement", "Main", "Second", "Garage"). Each has a name, an optional display order, and its own drawn plan + optional background image.
  - **Floor switcher** in the map view: a one-tap segmented control or slim tabs at the top showing all defined floors. Selected floor is rendered.
  - Components carry an optional `floor_id` (nullable until placed). A component can be on exactly one floor at a time. Moving a component between floors is supported.
  - Click-to-identify still works across floors: tapping a component in the Components list jumps to its floor + scrolls to its pin.
  - Existing per-panel `floor_plan_filename` migrates to a default "Main floor" row so existing data is not lost.

- **G14 — Components placed on drawn floor plans (G12 + G4 integration):** The existing place/drag/reposition UX (cycles 9–10) extends to the drawn-plan model:
  - The unplaced sidebar lists components without a `floor_id` (or without coords on the currently-displayed floor).
  - Dragging a component onto the drawn canvas creates `(floor_id, pos_x, pos_y)` for it.
  - Repositioning + Remove-from-map work identically to cycle 10.
  - Breaker-test workflow (G7) continues to work, but now respects floor scoping — "what just lost power on THIS floor" filters to the current floor.

### Post-G14 vision extension (cycle-15+)

After G11–G14 shipped, two structural problems surfaced once the app was actually used:

1. **Floors are a dead-end.** `MapLandingScreen` lets you create a floor, but renders the floors list as static `<span>`s — there's no way to open one. A floor only becomes visible on `PanelMapScreen` *if* a component on a panel's breakers already points at it. Fresh floors are effectively invisible.
2. **The mobile-only canvas isn't enough for "building a plan."** Drawing walls + rooms on a phone is fine for touch-ups, but the actual house-mapping task wants a desktop-class editor — big canvas, pan, zoom, keyboard shortcuts.
3. **Visual design feels rough.** G11's dark-mode + sky accent is functional but reads as a hacker prototype, not a polished product.

The vision is extended with three new goals to fix these:

- **G15 — Desktop-class map builder.** A standalone `/floors/:floorId/edit` route designed primarily for laptop/desktop use (still degrades to phone). Big canvas, dedicated layer/tool/properties sidebars, pan + zoom (mouse wheel + pinch), keyboard shortcuts (V = pointer, W = wall, R = room, ESC = cancel, Delete = remove selected). Continues to use the existing walls + rooms vector model. **The phone PWA primary surface stays touch-only; this is the "I'm at my laptop doing real planning" surface.**

- **G16 — Floor-management end-to-end UX.** Fix the dead-ends:
  - `MapLandingScreen` floor rows are tappable Links that open the floor's editor (or its panel-scoped map view).
  - "Create floor" navigates straight into the editor for the new floor — no separate step required.
  - Floor rename + reorder + delete UI lives on the same screen (a row context menu or inline edit).
  - A floor with zero components still shows up on `PanelMapScreen`'s switcher when its `floor_id` is referenced by ANY component on this panel's breakers OR explicitly via a `?floor=<id>` deep link.

- **G17 — Design-system v2 (visual refresh).** The cycle-11 token set stays — **token NAMES are pinned in CLAUDE.md and must not change** — but the *values* are reworked for a cleaner, friendlier feel:
  - New color ramp (proposed: warmer dark palette with a single high-contrast accent; or a light/dark dual-mode that respects `prefers-color-scheme`). Council picks.
  - Tighter type ramp, better leading, improved icon weight.
  - Real visual hierarchy on every screen (large-then-medium-then-supporting, instead of "all sections look the same").
  - Polish the primitives (Card, Button, Input) to feel less utilitarian.
  - **AA contrast still required.** Token *NAMES* preserved; only values change so screens don't need rewrites.

### Post-G17 vision extension (cycle-18+)

After G17 shipped, the user surfaced two more issues from actual use:

1. **Clicking on a floor "always just loads with skeleton."** The `FloorEditScreen` mounted via the AppShell-escape-hatch route hangs at the loading state under React StrictMode (Vite's dev default). Diagnosed: the `aliveRef = useRef(true)` pattern has a cleanup-only `useEffect` that sets aliveRef → false on unmount but never re-sets it to true on the second StrictMode mount, so `if (!aliveRef.current) return` short-circuits forever and `setLoading(false)` never fires. Regression in cycle 15. Counts as a **must-fix** under G16 ("no dead-end") and incidentally also G15.

2. **"I want a real view of my electrical panel upon building it, with vertical and horizontal config."** Real breaker panels have a visual layout — column-grid of numbered slots, with main breakers, double-wide breakers spanning two slots, half-height tandem breakers, etc. The current `PanelDetailScreen` shows a flat sorted list which doesn't match the user's mental model of the panel they're standing in front of. The user wants to see the panel as it physically exists.

- **G20 — UI polish + drag-to-link gangs.** Three improvements after using G19:

Modal + ConfirmModal + PromptModal + Picker variants; ESC-to-close, overlay-click-to-dismiss, focus trap, framed in the existing token system. `useModal` hook exposes imperative `confirm()` / `prompt()` / `pick()` so callers get Promise-returning ergonomics with no JSX gymnastics.

With a component tool active, a tap on the canvas creates that component AND places it at the tapped position in one action — no separate trip to the Components tab. The component gets an auto-generated name (e.g. "Outlet 4") which the user can rename via the properties sidebar.
  - **Multi-gang switches.** A "switch" component gains a `gangs` count (1–8; default 1). Each gang independently controls one or more lights/outlets. Schema: `components.gangs INTEGER NOT NULL DEFAULT 1`; new `switch_controls` table `(switch_id, gang_index, controlled_id)` with composite ON DELETE CASCADE. Properties sidebar shows each gang as a row with a multi-picker for which components it controls; the controlled lights/outlets show a "controlled by <switch> gang N" badge. Test mode (G7) is unaffected this cycle — gang-flip simulation is deferred.

- **G18 — Real panel visualization (vertical/horizontal).** PanelDetailScreen renders a visual diagram of the panel:
  - Panel gains an `orientation` field (vertical = tall, slots stacked top-to-bottom in two columns; horizontal = wide, slots laid out left-to-right). Vertical default.
  - Panel gains a `slotCount` field (total slot positions; typical residential 20 / 24 / 30 / 40 / 42).
  - Each breaker slot renders in its `slotPosition` (or its `slot` label mapped to an integer if numeric). Slots with no breaker render as empty placeholders ("spare").
  - Double-pole breakers (`poles: 'double'`) span TWO adjacent slot positions visually.
  - Tap a slot → opens the existing edit-breaker flow (or "Add breaker in slot N" if empty).
  - The existing flat list stays as a fallback/list-view toggle so power users don't lose anything.
  - Out of scope this goal: AFCI/GFCI badges, amp-rating color-coding, dead-front photos. Simple visual grid is the bar.

## Non-goals (drift guard — meta-ralph will reject proposals that try to add these)

- Multi-user collaboration, sharing, or accounts. Single-user only.
- Cloud sync, SaaS subscriptions, or any required public-internet service. The stack runs on the user's hardware.
- IoT / smart-home integration.
- Real-time electrical monitoring, current sensing, or hardware integration.
- Code-compliance auditing, load calculations, wire-gauge advice.
- Permit paperwork, inspection workflows, professional contractor features, or multi-property portfolio management.
- AI / ML features.
- Mobile-native apps (PWA is sufficient).
- Desktop UI as a goal. (The PWA may degrade gracefully on a wide viewport for dev convenience, but desktop polish is incidental.)
- Off-network usage when the server is off. Brief offline tolerance via cache is welcome.
- Kubernetes / Swarm / multi-node / Helm.
- Host-OS-specific installers (no MSI, no .deb, no Homebrew).
- **CAD-like precision floor planning** — G12 is a sketching tool for "approximately where the walls are," not AutoCAD. No dimensioning, no scale calibration, no DXF/SVG import beyond simple line+rectangle primitives.
- **A separate floor-plan format with proprietary semantics** — `walls` and `rooms` are simple line/polygon records in Postgres; they're not a separate file format the user has to learn or back up separately.

### Cycle-42 vision extension (user-directed, post-research synthesis)

After the cycle-42 research synthesis (codebase audit + UX walkthrough + commercial-app feature research), the user picked: ship **everything except Photos** AND added one NEW concrete data-model ask: tandem breakers should be modeled as two independent circuits ("6a" and "6b"), since electrically that's what they are. The tandem ask is a data-model change that **gates** the slot-validation work (G33) and the impact view (G35) — so it ships FIRST. The rest of the batch then sequences by value × independence.

- [x] G34 — Tandem breakers = two independent circuits (cycle-42). User quote: "tandem breakers need to be treated as 6a and 6b for example, cause they are really 2 circuits in the end." Today a tandem is ONE breaker row with `poles='tandem'` and a 45° stripe pattern in the viz — incorrect: a tandem stuffs two single-pole breakers into one stab, so it's TWO circuits at the same slot N, addressed as "Na" and "Nb". Data model: add `breakers.tandemHalf TEXT CHECK (tandemHalf IN ('a','b') OR tandemHalf IS NULL)`; tandem breakers carry slotPosition=N + tandemHalf='a' or 'b'; single/double leave tandemHalf NULL. Slot validation (G33 `validateSlotAssignment`) updates: a slot N may host EITHER (one non-tandem breaker, tandemHalf NULL) OR (up to two tandem-half breakers, one 'a' + one 'b'). Display: slot label "Na" / "Nb" for tandems; just "N" otherwise. Panel viz: tandem cell splits into two half-height sub-cells (vertical orientation: a above b; horizontal: a left of b). BreakerForm: when poles=tandem, a "Half" picker (a/b) appears; validation rejects creating a 'b' if 'a' doesn't exist on the same slot yet (and vice-versa). Print + audit screens render tandem halves as separate rows. Migration: existing tandem breakers (no `tandemHalf` yet) get `tandemHalf='a'` so they remain valid; a follow-up UI prompt nudges users to create the 'b' partner if any.

- [x] G35 — Inverse impact view + critical-circuit flag (cycle-42+). One-tap "what dies if I flip this breaker" view for power-outage triage. Each breaker row gains an "Impact" button that opens a modal listing every component the breaker controls (directly via `components.breakerId` AND transitively via `switch_controls` — switches on this breaker pass through to the lights/outlets they control). Components grouped by floor + room, sorted by criticality. Add `components.critical BOOLEAN NOT NULL DEFAULT 0` — UI toggle on each component card (fridge / freezer / well pump / sump pump / modem / medical etc.). Critical items render with a warn-orange badge + sorted to the top of the impact list. Once G39 (subpanels) ships, impact recurses through `panels.parent_breaker_id` chains automatically.

- [x] G36 — Breaker-test audit trail (cycle-42+). G7's walk-through workflow is ephemeral today — "I verified slot 14" leaves no trace. ## Constraints (technical, legal, operational)

- **Deployment unit:** single `docker-compose.yml` with two services — `app` (a single Node/Hono process serving the API, uploaded floor-plan images, AND the built PWA) and `db` (Postgres). One host port exposed (`HOST_PORT=8070` by default). No in-stack TLS — HTTPS is handled by the operator's external reverse proxy (Caddy / Cloudflare Tunnel / etc.). This matches the user's HousesTracker deployment pattern. (The original cycle-33 backend+nginx split was collapsed into the single `app` image; see CLAUDE.md "Single-image consolidation".)
- **Server host:** any Docker-capable Linux / Windows / macOS machine. Images target `linux/amd64` + `linux/arm64`.
- **Frontend host:** the user's phone. Installable as PWA.
- **Network:** LAN-only or via a tunnel. No port-forwarding required; no public internet exposure assumed.
- **TLS / HTTPS:** terminated by the operator's external proxy, not by the stack. The compose stack serves plain HTTP on the host port.
- **Auth:** single-user login gate (feat/auth-gate). First-time sign-up screen on empty DB; scrypt-hashed password in Postgres; 30-day JWT cookie. Trust boundary stays the LAN or the operator's reverse proxy / tunnel — auth is "casual visitor" deterrence, not "internet-exposed hardening".
- **Database:** PostgreSQL (the `db` compose service), relational data in a named Docker volume. Floor-plan images + the `.auth-secret` live in a separate bind-mounted directory.
- **Licenses:** MIT / Apache-2 / BSD / ISC dependencies only. No GPL.
- **Touch:** all interactive targets ≥44×44 CSS px.
- **Resilience:** brief loss of LAN connectivity should not crash the PWA. Cached last-known data with a clear "not connected" indicator.
- **Image hygiene:** multi-stage builds, alpine or distroless base, non-root user inside containers, no secrets in images, config via env vars.
- **Performance budget (added by G11):** Initial PWA load (cold cache) ≤ 200 KB gzipped JS. No layout shift after first paint. No interaction takes more than one animation frame (16ms) to begin responding visually. The drawing canvas (G12) targets 60 fps pan + zoom on a 2020-era mid-range Android phone.

## Out-of-loop decisions (the meta-loop is not allowed to change these without human review)

- **Two-tier architecture:** phone-side PWA + server-side Node backend in a container. A single Node/Hono process serves the static PWA bundle + `/api` + `/files`; a sibling Postgres container holds the data. External proxy for HTTPS.
- **Frontend = PWA.** Not Electron, Capacitor, native, or desktop-only.
- **Backend persistence = PostgreSQL** (the `db` compose service) in a named Docker volume; floor-plan images + `.auth-secret` in a bind-mounted directory.
- **Single-user, LAN-only** (or behind operator's tunnel) + a minimal login gate (feat/auth-gate: sign-up on first boot, scrypt-hashed password in Postgres, JWT cookie). No multi-user, no roles, no signup beyond the first user.
- **Mobile-first.** Desktop browser is incidental.
- **Docker Compose is the deployment surface.** No K8s / Swarm / Helm / host-package-manager installers.
- **No in-stack TLS** (locked in by the HousesTracker-pattern conversion). HTTPS is external. Operator picks the tool.
- **Floor plans are vector data, not bitmaps** (G12). The image-upload path stays as an *optional* underlay.
- **Floors are house-level entities** (G13), not per-panel. Panels stay panel-level (a panel doesn't have its own floor — but its breakers' components do, indirectly, via their `floor_id`).
- **Drawing primitives are minimal**: line segments (walls) + axis-aligned rectangles or polygons (rooms). No curves, no doors, no windows, no furniture in the initial G12 scope. Those are future-vision-extension topics if the user wants them.
